本站现由 Maverick 强力驱动

熊猫小A 2019-12-08 in 偶尔Geek

好，我终于还是转向了静态博客。其实这个打算很早就有，实际上我年初还专门写了把 Typecho 导出为 Markdown 文件的插件。写出来自然是为了用，但没想到快一年了才终于用上。

与之前几次无疾而终的尝试不同，这次我自己写了一个静态网站生成器，名叫「Maverick」。它使用 Python 编写，能从指定的目录中自动搜寻 Markdown 文件，并根据与 Hexo 相似的 YAML frontmatter 生成静态网站。Maverick 附带一款简洁的默认主题「Galileo」，二者结合，成为了我相当喜欢的一款作品。现在，「无文字」与「无知识」都使用 Maverick 驱动并部署于 Coding Pages 上，同时依旧由又拍云提供 CDN 支持。

虽然我很想解释一下自己为什么要进行这次转变，以及为什么要自己写生成器，但思来想去并没有什么决定性的原因。也许是我觉得管理动态博客过于繁琐？也许是我单纯想写写 Python？

Alan Key 的一句名言，被 Steve Jobs 引用后众人皆知：

People who are really serious about software should make their own hardware.

拿来活用，则「任何认真对待博客的人都应该自己写博客系统」。这话说起来固然偏执，但确实为吹毛求疵、喜欢不断追求的人指了一条明路。既然现在的轮子都不喜欢，那就应该造自己的轮子。

现有的生成器主要有三点让我不满。一者，它们容易变得很慢；其次，它们处理源文件（文章、图片等）的方式并不符合我的需求；另外，我很难在它们的基础上做更改，修修补补不好玩。速度方面我没有做什么分析，个人感觉相比 Hexo 还是要快的。主要令我满意的是生成逻辑与针对我自己需求的改善。

Maverick 支持从任何目录中递归地搜寻 Markdown 文件，因此它不限制把文章存在哪里，也不限制里面的目录结构，文章类型、分类、标签等都完全由 frontmatter 决定；另外，它对图片的处理有一项重要改进：支持发布以任何方式引入的图片，本地相对路径、本地绝对路径、网络图片都可以，最终 Maverick 会将所有的本地图片集合到一起，以统一的方式发布到静态网站中；针对远程图片，还可以在生成时将它们拉取到本地并缓存，以同样的方式发布到静态网站中。这么设计的原因主要有：

  1. 如此可以在本地的任何编辑器中方便地插入与预览图片。不论是 Typora、VS Code 还是什么，都支持以本地相对路径或绝对路径引入的图片，由此实现所见即所得，不用操心图片上传什么的；在未来完全可以实现自动的图片压缩。而现有的生成器一般不能处理本地图片，或者处理起来很丑陋；
  2. 插入本地图片让 Maverick 可以提前明确图片尺寸，为照片排版、懒加载等功能提供了先验条件；
  3. 虽然目前还没有这个功能，但这样的图片发布方式为 CDN 支持提供了有利条件。

Maverick 的另一项改进是文章解析这一块。以 Hexo 为例，默认的解析器对脚注、照片排版、特殊标注等支持总是缺斤少两，而 Maverick 使用 Lepture 开发的 Markdown 解析器 mistune，并由我进行扩展，一开始就是按照自己的需求写的，因此不存在这些问题。实际上，在文章内容方面 Maverick 与 VOID 一脉相承，完美支持图片排版、notice、友链、行内脚注、Ruby 注音等语法，另外还有无需 JS 的代码高亮。在此之上，还支持数学公式、DPlayer，以及针对 Bilibili 播放器的优化。请前往常用页面元素示例查看效果。Maverick 原生支持生成 RSS 与 Sitemap。并且，还支持与 ExSearch 相同的实时搜索。

默认的主题名叫 Galileo，不是我的原创设计，而是「安妮薇邮报」、hexo-theme-cactus 的混合物，内容区域则是我非常喜欢的 yue.css。我实在是没有精力再设计一款主题了。

关于 Maverick 大致就是这些。其实还有许多细节我十分喜欢，不过我自己知道就行了，说出来比较枯燥。

对了，虽然我很想推荐你们尝试一下 Maverick，但必须提醒：Maverick 只是我个人花费了几天写的东西，不成熟、也不稳定。实际上目前是很不成熟、很不稳定。没有文档，各项配置也没什么注释。如果你没有接触过静态博客，并且没有基本的 Python 知识，用起来恐怕会很痛苦。

那为什么还要开源呢？当然是为了骗⭐啊笨蛋！我明年能不能找到工作可就靠你们这几个 star 了！快去给我点赞吧！~
